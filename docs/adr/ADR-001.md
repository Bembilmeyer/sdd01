# ADR-001: Контейнерный стиль приложения

- **Статус:** принято
- **Дата:** 2026-06-30
- **Владелец решения:** заказчик
- **Связь с VISION:** D1-F1, D1-F2

## Контекст

Нужно выбрать базовый контейнерный стиль для приложения поиска по корпоративной базе знаний. Решение должно поддержать быстрый вывод первой версии, минимальный UI, логин сотрудника, генерацию ответа на естественном языке и возможность в будущем подключать новые источники знаний.

## Рассматриваемые варианты

| Вариант | Trade-off | При каком условии выигрывает |
|---------|-----------|------------------------------|
| Модульный монолит | Минимальная операционная сложность и быстрый старт, но меньше независимого масштабирования частей | Если приоритетом является быстрый запуск первой версии и простота сопровождения |
| UI + API + отдельный RAG-сервис | Лучше изоляция AI-контура и проще отдельно масштабировать поиск/генерацию, но выше сложность интеграции и деплоя | Если ожидается быстрый рост нагрузки и отдельное развитие AI-контура |
| Набор микросервисов | Максимальная гибкость и изоляция, но самая высокая стоимость разработки и эксплуатации | Если есть жёсткие требования к изоляции контуров или независимым командам |

## Решение

Заказчик выбрал модульный монолит как стартовый контейнерный стиль приложения.

## Обоснование

Это решение лучше соответствует текущему VISION: критичны скорость запуска, простота интерфейса и возможность быстро сократить время обработки обращений. Требование D1-F2 по подключению новых источников можно закрыть через внутренний модуль адаптеров без раннего дробления на сервисы.

## Отвергнутые варианты и почему

> Самая ценная часть ADR — не теряйте её.

- **UI + API + отдельный RAG-сервис** — отвергнут, потому что на старте даёт дополнительную сложность деплоя и наблюдаемости без подтверждённой необходимости.
- **Набор микросервисов** — отвергнут, потому что для начального этапа это избыточно и увеличивает стоимость изменений.

## Последствия

Архитектура будет развиваться как один деплоимый backend с внутренними модулями `search`, `answer-generation`, `session-analytics`, `source-connectors`. Нужно отдельно принять решения по конкретному backend-стеку, UI-стеку, хранилищам и security-контуру.
