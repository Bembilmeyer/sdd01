# ADR-004: Хранилище поиска и сессионных данных

- **Статус:** принято
- **Дата:** 2026-06-30
- **Владелец решения:** заказчик
- **Связь с VISION:** D1-F1, D1-F2

## Контекст

Нужно выбрать хранилище для поискового индекса и сессионных данных. Решение должно поддерживать retrieval для ответа на естественном языке, сохранение поисковых сессий и возможность постепенно подключать новые источники знаний.

## Рассматриваемые варианты

| Вариант | Trade-off | При каком условии выигрывает |
|---------|-----------|------------------------------|
| PostgreSQL + pgvector | Простой старт, один технологический контур для OLTP и векторного поиска, но меньше специализированных search-возможностей | Если приоритет простота эксплуатации и умеренный масштаб старта |
| Elasticsearch / OpenSearch + отдельный векторный контур позже | Сильный полнотекстовый поиск и развитая search-аналитика, но больше инфраструктуры и сложнее первая версия | Если полнотекстовый поиск важнее простоты |
| Специализированное векторное хранилище | Сильнее для сложного semantic retrieval, но требует отдельной инфраструктуры и повышает стоимость старта | Если retrieval-сценарии сразу сложные и объём данных велик |

## Решение

Заказчик выбрал `PostgreSQL + pgvector` как общее хранилище для сессий, аналитики и поискового индекса.

## Обоснование

Решение минимизирует число инфраструктурных компонентов для первой версии и хорошо сочетается с выбранным модульным монолитом. Это поддерживает D1-F1 через единый контур поиска и D1-F2 через более дешёвое расширение источников на старте.

## Отвергнутые варианты и почему

> Самая ценная часть ADR — не теряйте её.

- **Elasticsearch / OpenSearch + отдельный векторный контур позже** — отвергнут, потому что добавляет лишнюю инфраструктурную сложность на старте.
- **Специализированное векторное хранилище** — отвергнут, потому что пока нет подтверждённой потребности в отдельном высокоспециализированном контуре retrieval.

## Последствия

Понадобится спроектировать схему PostgreSQL так, чтобы разделить транзакционные таблицы сессий и таблицы индекса/эмбеддингов. Если объём документов или требования к поиску вырастут, возможен последующий вынос retrieval-контура в отдельное search-хранилище.
