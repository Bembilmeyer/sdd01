# ADR-003: UI-стек desktop-приложения

- **Статус:** принято
- **Дата:** 2026-06-30
- **Владелец решения:** заказчик
- **Связь с VISION:** D1-F1

## Контекст

Нужно выбрать UI-стек для desktop-версии приложения. По VISION интерфейс должен оставаться минимальным: ввод поискового запроса, просмотр результатов, логин и логаут сотрудника.

## Рассматриваемые варианты

| Вариант | Trade-off | При каком условии выигрывает |
|---------|-----------|------------------------------|
| ASP.NET Core MVC / Razor Pages | Самый простой стек в рамках .NET, простой деплой и интеграция с аутентификацией, но ограниченная гибкость для сложного интерактивного UI | Если приоритет простота и быстрый корпоративный запуск |
| Blazor Server | Единый .NET-стек и меньше JavaScript, но своя модель соединений и операционные особенности | Если нужен более интерактивный UI без отдельного SPA |
| React + ASP.NET Core API | Самая гибкая модель UI, но больше технологических слоёв и выше сложность сборки/деплоя | Если ожидается быстрое усложнение интерфейса |

## Решение

Заказчик выбрал `ASP.NET Core MVC / Razor Pages`.

## Обоснование

Решение соответствует non-goal из VISION: не делать сложный UI. Основная сложность продукта находится в поиске и генерации ответа, а не во фронтенде, поэтому упрощение UI-стека снижает стоимость первой версии.

## Отвергнутые варианты и почему

> Самая ценная часть ADR — не теряйте её.

- **Blazor Server** — отвергнут, потому что не даёт решающего преимущества для минимального UI, но добавляет архитектурные особенности, которые пока не нужны.
- **React + ASP.NET Core API** — отвергнут, потому что это лишняя сложность для текущего объёма интерфейса.

## Последствия

UI будет рендериться сервером в рамках монолита. Если в будущем интерфейс усложнится, возможен пересмотр в пользу отдельного SPA, но на старте это не требуется.
